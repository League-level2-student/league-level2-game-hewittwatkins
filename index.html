2048 Version 2.0

by: Hewitt Watkins

This game was made for my level 2 project at The League of Amazing Programmers.
Click here to download and play!

1. Why I Made This Game

I decided to make this game because I wanted to improve upon 2048's game logic. I also wanted to make the game more interesting by forcing players to think about their moves and make 'randomly pressing the arrow keys' an invalid game strategy. I was also excited to add an appealing dark themed color scheme.

2. What I Learned

The two main takeaways I learned from user testing from teachers, classmates, and friends was that the game was originally too easy and that it needed instructions. Before testing, the player had 3 lives. However, after watching a couple of my friends reach 2048 in my game without too much trouble, I altered the game to only let the user have 1 life. Furthermore, I also found my game logic changes to the game weren't inherently obvious. My peers couldn't tell what changes I had made to make the game more fun. As a result, I added some information about that on the home page.

3. Architecture of My Game

The easiest way to think about my game's design is to imagine a board. This is the 2048 board and pretty much the GamePanel class. On this board, there are then squares or tiles. These are from the NumSquare class and are stored in a 2D array that simulates the 4 by 4 layout of the board. The methods in GamePanel and NumSquare are combined to complete moves on the board.

4. Technical Implementation

The hardest part of implementing my game was figuring out how to check when the game was over. This was more complicated than I expected because I had to create a method for each type of movement (up, down, left, right) when I thought that I could use one method for all. In order to solve this problem, I coded the logic for one movement and tried to use it for all 4 moments. As you can expect, all arrow keys then only produced one move. Eventually I caught on and created 3 more movement methods (one for each arrow key). Overall, it was the trial and error method that helped me achieve my goal. Frequent testing was key.

5. Most Challenging Part

The most difficult part of this project for me was the graphics. I found it hard to imagine the positions of numbers and squares on the board because the x-y axes for graphics start in the top left (different from what I am used to). To overcome this challenge, I created a function that used each square's position on the board in the 2D array to draw itself on the GUI. This made the process very easy because I only had to do the math for one square and then throw it into a function.

6. Future Fixes and Updates

The only improvement I would like to make on my game at the movement is to make the square movements animated. At the moment, they are re-drawn to their new position immediately, which does not affect the playability of the game. However, it does not look as nice as I would like. This is certainly something I will look into in the future.
